# DNS Shop Microservices

Учебный проект имитации магазина DNS на микросервисной архитектуре. Все компоненты собираются и запускаются внутри **Docker**.

## Структура проекта

*   **user-service** (порт 8080) — управление пользователями
    *   **Фичи**: Регистрация с профилем в одной транзакции, **Optimistic Locking** (@Version), интернационализация ошибок (i18n), частичное обновление (PATCH), мапперы с игнорированием `null`.
*   **product-service** (порт 8082) — каталог товаров и категорий.
    *   **Фичи**: Реактивный **JOIN** (Товар + Категория), гибкие характеристики через **JSONB**, динамические фильтры через `DatabaseClient`, пагинация, **аналитика фильтров** (подсчет количества товаров по атрибутам внутри JSONB).
* **cart-service** (порт 8083) — управление корзиной покупок.¶
* **Фичи**: Сборка ответа через **Batch-запрос** к каталогу (исключение N+1), умный **UPSERT** (инкремент количества), универсальный **PATCH**.¶

## Эндпойнты системы

### User-Service


| Метод | Путь | Описание |
|-------|------|----------|
| `GET` | `/api/v1/users` | Получить список всех пользователей по фильтру |
| `GET` | `/api/v1/users/{id}` | Получить полную информацию о пользователе (User + Profile) |
| `POST` | `/api/v1/users` | Регистрация нового пользователя |
| `PATCH`| `/api/v1/users/{id}` | Частичное обновление профиля (игнорирование null) |

### Product-Service


| Метод | Путь | Описание |
|-------|------|----------|
| `GET` | `/api/v1/products` | Поиск товаров с пагинацией и фильтрами (цена, имя, JSON-атрибуты) |
| `GET` | `/api/v1/products/{id}` | Получить товар с названием категории и атрибутами |

### Cart-Service

| Метод | Путь | Описание |
|-------|------|----------|
| `GET` | `/api/v1/cart` | Получить корзину с полными данными о товарах |

## Архитектурные решения

*   **Идентификация (X-User-Id)**: Сервисы доверяют **Gateway**. `userId` извлекается из заголовка, что исключает лишние запросы к `user-service`. Мы предполагаем, что заголовок приходит валидным после авторизации.
*   **Межсервисное взаимодействие**: Для получения данных о товарах в корзине используется реактивный `WebClient`. Чтобы избежать N+1, запросы выполняются батчем, а данные объединяются в памяти через `Map`.
*   **REST Semantics**:
    *   `POST` используется для создания или "умного" инкремента.
    *   `PATCH` используется для изменения существующего ресурса, даже если бизнес-логика приводит к его удалению (при `quantity=0`).

## Тестирование

*   **Интеграционные**: Настроен **Singleton Testcontainers**, статический запуск для стабильности соединений, автозаполнение и очистка таблиц перед каждым тестом.
*   **Параметризованные**: При тестировании фильтрации, валидации и добавления товаров в корзину.
*   **Изоляция (WireMock)**: Используется для имитации `product-service`. Это исключает зависимость тестов корзины от состояния каталога.
*   **Jacoco**: Покрытие бизнес-логики > 80%.

## План развития (Backlog)

### Общее
1.  **Gateway & Security**: Внедрение Spring Cloud Gateway для централизованной проверки JWT и подстановки `X-User-Id`.
2.  **Order-Service**: Реализация сервиса заказов.
3.  **Messaging**: Внедрение **Kafka/RabbitMQ**. Метод `clearCart` лучше вызывать через брокер, чтобы процесс оформления заказа не зависел от доступности сервиса корзины.

### Product-Service
1.  **Улучшение поиска**: Поддержка диапазонов (gt/lt) для числовых характеристик внутри JSONB.
2.  **Оптимизация фильтров**: Внедрение фоновых **Jobs** для пересчета счетчиков фильтров при больших объемах данных.

### Cart-Service
1.  **Persistence Strategy**: Рассмотреть перенос хранения временных корзин в **Redis**, оставив PostgreSQL только для долгоживущих данных.
2.  **Caching**: Кэширование данных о продуктах (имена, цены) для снижения нагрузки на `product-service`.
