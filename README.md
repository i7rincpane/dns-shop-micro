# DNS Shop Microservices

Учебный проект имитации магазина DNS на микросервисной архитектуре.  

* Проект реализован как **мультимодульное Gradle-приложение**, где каждый сервис является независимым микросервисом со своей базой данных.
* Все компоненты собираются и запускаются внутри **Docker**.
* **API Documentation (Swagger/OpenAPI)**: Спецификации в формате YAML выгружаются в директорию `docs/openapi/`.

## Структура проекта

*   **user-service** (порт 8080) — управление пользователями.
    *   **Фичи**: Регистрация с профилем в одной транзакции, **Optimistic Locking** (@Version), интернационализация ошибок (i18n), частичное обновление (PATCH), мапперы с игнорированием `null`.
*   **product-service** (порт 8082) — каталог товаров и категорий.
    *   **Фичи**: Реактивный **JOIN** (Товар + Категория), гибкие характеристики через **JSONB**, динамические фильтры через `DatabaseClient`, пагинация, аналитика фильтров.
*   **cart-service** (порт 8083) — управление корзиной покупок.
    *   **Фичи**: Сборка ответа через **Batch-запрос** к каталогу (исключение N+1), умный **UPSERT** (инкремент количества), универсальный **PATCH**, асинхронная очистка через Kafka.
*   **order-service** (порт 8084) — оформление заказов и оркестрация.
    *   **Фичи**: Transactional Outbox, реактивное взаимодействие с другими сервисами, локальная компенсация остатков.

## Архитектурные решения


### Cart-Service

*   **Идентификация (X-User-Id)**: Сервисы доверяют **Gateway**. `userId` извлекается из заголовка, что исключает лишние запросы к `user-service`. Мы предполагаем, что заголовок приходит валидным после авторизации.
*   **Межсервисное взаимодействие**: Для получения данных о товарах в корзине используется реактивный `WebClient`. Чтобы избежать N+1, запросы выполняются батчем, а данные объединяются в памяти через `Map`.
*   **REST Semantics**:
    *   `POST` используется для создания или "умного" инкремента.
    *   `PATCH` используется для изменения существующего ресурса, даже если бизнес-логика приводит к его удалению (при `quantity=0`).

### Order-Service

*   **Контроль остатков и борьба с Race Condition**: Используем **пессимистический подход** на уровне SQL-запроса: `UPDATE products SET quantity = quantity - :qty WHERE id = :id AND quantity >= :qty`. Это гарантирует, что количество товаров никогда не станет отрицательным даже при сотнях одновременных запросов на один и тот же товар.
*   **Transactional Outbox**: Для гарантированной доставки сообщений в Kafka используется паттерн Outbox. Событие для Kafka сохраняется в ту же БД, что и заказ. Если база упала — событие не создастся. Если заказ создался — событие гарантированно будет отправлено позже - `At-least-once` . Но могут быть дубли (при повторной попытке (Retry), если транзакция пройдет, но сервис из за обрыва и таймаута не получит ответ) и тут должен помочь UUID сгенерированный до начала транзакции)
*   **Snapshot данных (Заморозка цен)**: В момент оформления заказа название товара и его цена фиксируются в таблице `order_items`. Это гарантирует, что изменение цены в каталоге (product-service) не изменит стоимость уже созданного заказа.
*   **Event-Driven (Хореография)**: Очистка корзины вынесена в асинхронный процесс. `order-service` только публикует событие, а `cart-service` самостоятельно реагирует на него. Это снижает время отклика для пользователя и повышает отказоустойчивость. Операция идемпотентная сама по себе (второй раз удалять нечего).
*   **Локальная компенсация**: В методе создания заказа встроена мгновенная реакция на ошибки. Если запись заказа в БД не удалась после успешного списания склада, сервис немедленно вызывает возврат товара (`increaseStock`) через WebClient.
*   **Децентрализованный UUID в Outbox**: Использование UUID вместо БД-сиквенса дает два ключевых преимущества:
  1. **Производительность**: ID генерируется в памяти процессора (наносекунды) без обращения к базе за следующим значением. Это убирает лишнее ожидание ответа от БД в реактивной цепочке.
  2. **Уникальность при интеграции**: UUID гарантирует, что события из разных систем (или даже разных инстансов одной системы) никогда не пересекутся по ID. Это критически важно при объединении данных в общем логе или аналитике, где сиквенсы из разных баз неизбежно начали бы конфликтовать.

## Тестирование

*   **Интеграционные**: Настроен **Singleton Testcontainers**, статический запуск для стабильности соединений, автозаполнение и очистка таблиц перед каждым тестом.
*   **Параметризованные**: При тестировании фильтрации, валидации и добавления товаров в корзину.
*   **Изоляция (WireMock)**: Используется для имитации `product-service`. Это исключает зависимость тестов корзины от состояния каталога.
*   **Jacoco**: Покрытие бизнес-логики > 80%.

## План развития (Backlog)

1. **Gateway & Security**: Внедрение Spring Cloud Gateway для централизованной проверки JWT и подстановки `X-User-Id`.
2. **Saga Pattern (Оплата)**: Внедрение `payment-service`. Реализация логики отмены заказа и компенсации склада при получении события `PAYMENT_FAILED`.
3. **Order TTL (Авто-отмена)**: Реализация фоновой задачи (Job) для автоматической отмены заказов, не оплаченных в течение 30 минут, с возвратом зарезервированных товаров на склад.
4. **Distributed Tracing**: Внедрение Trace ID для отслеживания пути запроса через все микросервисы и брокер сообщений.
5. **Debezium (CDC)**: Переход от `OutboxScheduler` к Debezium для стриминга событий напрямую из WAL-логов Postgres в Kafka. Это позволит убрать фоновые опросы БД и снизить нагрузку.
6. **Идемпотентность на стороне базы**: Вынос генерации UUID в `order-Service` до начала транзакции для реализации полной идемпотентности на стороне базы(невозможность повторно создать два события с одинаковым UUID).
7. **Идемпотентный потребитель (Idempotent Consumer)**: Внедрение проверки UUID сообщения на стороне получателей (например, в `cart-service`). Использование Redis или таблицы в БД для хранения ID обработанных событий позволит игнорировать повторные сообщения. Переотправки в режиме `At-least-once` неизбежны, если в момент отправки подтверждения (Ack) от Кафки или от БД Шедулеру происходит сбой сети. В этом случае Шедулер отправит сообщение повторно, так как не получил гарантии успешной доставки, хотя сообщение на самом деле могло уже попасть в топик.
8. **Улучшение поиска**: Поддержка диапазонов (gt/lt) в `product-Service` для числовых характеристик внутри JSONB.
9. **Оптимизация фильтров**: Внедрение фоновых **Jobs** в `product-Service` для пересчета счетчиков фильтров при больших объемах данных.
10. **Persistence Strategy**: Рассмотреть в `cart-Service` перенос хранения временных корзин в **Redis** (с установкой времени жизни записей - ТТL), оставив PostgreSQL только для долгоживущих данных.
11.  **Caching**: Кэширование данных в `cart-Service` о продуктах (имена, цены) для снижения нагрузки на `product-service`.
